## ⚡ OLTP vs OLAP

| | OLTP | OLAP |
|---|---|---|
| 目的 | 跑业务 | 做分析 |
| 操作 | CRUD | 几乎只读 |
| 数据量 | 行级 | 百万~亿行 |
| 代表 | MySQL/Oracle 业务库 | Snowflake/Redshift/SSAS |

## 🔍 OLAP 五大操作

| 操作 | 含义 | 类比 |
|------|------|------|
| **Drill-Down** | 下钻 | 年→月→日，越来越细 |
| **Roll-Up** | 上卷 | 日→月→年，越来越粗 |
| **Slice** | 切片 | 只看"2024年"（固定一个维度） |
| **Dice** | 切块 | 2024年+华东+电子（固定多个维度） |
| **Pivot** | 旋转 | 行列互换，换个角度看 |

## 📖 Kimball 四步维度建模法

1. **选定业务过程** — 要分析什么？（销售、库存、物流）
2. **声明粒度** — 事实表一行代表什么？（一笔订单/一天汇总）
3. **确定维度** — 从哪些角度看？（时间、产品、门店、客户）
4. **确定事实** — 度量什么指标？（金额、数量、成本）

## 🧠 核心思维模型

```
业务系统（OLTP）          数据仓库（OLAP）
┌──────────────┐         ┌──────────────┐
│  快速写入    │  ETL    │  快速查询    │
│  单条记录    │ ──────→ │  批量聚合    │
│  规范化存储  │  搬运   │  反规范化    │
│  面向应用    │         │  面向主题    │
└──────────────┘         └──────────────┘
```

> 🔑 记住这一句话就够了：**OLTP 为"写"优化，OLAP 为"读"优化；ETL 是把"写优化"的数据变成"读优化"的数据的过程。**

## 🎯 常见面试问题

<details>
<summary>1. 什么是缓慢变化维 (SCD)？有哪几种类型？</summary>

**SCD (Slowly Changing Dimension)**：维度表中属性会随时间缓慢变化（如客户地址变了）。

- **Type 0**：不改，保留原值
- **Type 1**：直接覆盖（丢失历史）
- **Type 2**：新增一行，用时间戳标记（保留历史，最常用）
- **Type 3**：新增一列存旧值（只保留上一次）
- **Type 4**：拆成主表和历史表
- **Type 6**：Type 1 + 2 + 3 的混合
</details>

<details>
<summary>2. 星型模式和雪花模式怎么选？</summary>

**选星型**（99%的情况）：查询快，JOIN 少，维度表冗余可以接受。
**选雪花**（仅当）：存储极度紧张 + 维度属性频繁变更 + 规范化管理强制要求。
</details>

<details>
<summary>3. ETL 和 ELT 的区别？</summary>

- **ETL**：先转换再加载 → 数据进仓前已处理干净（传统做法）
- **ELT**：先加载再转换 → 原始数据先进仓，用数仓算力处理（云数仓主流）
</details>

<details>
<summary>4. 什么是数据湖？和数据仓库的区别？</summary>

- **数据仓库**：结构化、已处理、面向分析（"瓶装水"）
- **数据湖**：原始格式、未处理、面向存储（"水库"）
- **湖仓一体 (Lakehouse)**：两者融合，一个平台同时支持原始存储和分析查询
</details>
